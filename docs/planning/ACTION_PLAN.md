# 卒業研究：ToDoアクションプラン（現状→ゴール）

更新日: 2025-12-22

（追記）2025-12-31: ローカル（Ollama）でデモ完走、`duration` 記録も実装。

## 目的（ゴール）

- **個人のクローンAI**（Condition: P）と **汎用AI**（Condition: G）を比較し、参加者評価で差（Identity/Naturalness/Offensiveness）を検証する
- 実験で得たJSONデータを集計・分析し、**論文・発表**として提出できる状態にする

## 現状把握（実装ベース）

### できていること（主要機能）

- 実験UI（4セッション→評価→データDL）のフローが動く
- 条件（G/P）はクライアントでランダム割当
- チャット応答は Next.js API（/api/chat）で生成
  - G: 通常応答
  - P: RAG（ベクトルストア）から関連記憶を検索してコンテキスト付与
- Limitless → Markdown保存 → Embedding ingest（ベクトルストア生成）の経路が用意されている

### 未完了／リスク（実験実施に直結）

- セッション `duration` は実装済み（0固定は解消）
- P条件のRAGは **事前に ingest を回して vector-store.json を作る必要**がある（手順の明文化と再現性が必要）
- Ollama運用（インストール/起動/モデルpull/環境変数）のため、実験前に **手順の固定と動作確認**が必要
- 実験データの **集計・統計・可視化**（分析パイプライン）が未整備
- 新規: 応答品質（人間らしさ）が不足しているため、より高性能なローカルモデルの導入・選定が必要

## 最短で「実験できる状態」まで持っていく優先順位

### Milestone A（実験の動作保証）

- A1: ローカルで `experiment-ui` が安定稼働（Ollama導入・モデル揃い）
- A2: Condition P のRAGが機能（ingest後、/api/chat が retrieved_context を返す）
- A3: セッション所要時間の記録（duration算出）が入る
- A4: パイロット実験（身内1〜2名）で手順とデータが崩れないことを確認

追加（品質）

- A5: モデル品質の改善（候補モデル比較→本採用モデル決定→デモ再完走）

### Milestone B（本実験）

- B1: 被験者募集・同意・実施手順の確定
- B2: 目標人数分の実験を実施し、JSONを回収
- B3: 回収データの欠損・異常をチェックし、分析に回す

### Milestone C（分析→論文化）

- C1: 集計（条件別の平均/分散/箱ひげ等）
- C2: 統計（例: 条件G/Pの差の検定。手法は指導教員と合意）
- C3: 図表化・考察・限界・今後
- C4: 論文提出版＋発表スライド＋デモ手順

## レビュー運用（毎週ループ）

- 週1回: 「予定 vs 実績」「ブロッカー」「次週の最重要3つ」を更新
- 実験当日は: 失敗時の切り戻し（例: P条件が不安定ならGのみで最小実験）を事前に決める
